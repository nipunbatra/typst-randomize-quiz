quizforge

Write an exam as a plain Typst document. Get randomized sets, answer keys, and a grading CSV — deterministically, forever.

You write this

#show: quiz.with(id: "es335-quiz-1", course: "ES 335: Machine Learning",
                sets: ("A", "B", "C", "D"), answer-grid: true)

= Multiple Choice

+ #m(2) Increasing $k$ in $k$-NN typically has which effect?
  - ✓ Increases bias, decreases variance
  - Decreases bias, increases variance
  - None of the above          // auto-pinned last

= Fill in the Blanks

+ #m(1) Computing inner products in feature space without
  constructing features is the kernel #blank[trick].

…compiled as-is it renders the answer key; each --input set=… is a different randomized paper.

Students get these

Why it's safe to trust with a real exam

  • Deterministic. Sets are pure functions of (quiz id, set id, question) — rebuilds are byte-identical. No clocks, no OS randomness, no dependencies.
  • Fair by construction. Every set has the same questions and total marks; only order differs. Bank sampling can't diverge across sets.
  • Keys can't drift. Paper, key PDF, and grading CSV derive from one structure — and the build re-verifies every letter↔option mapping.
  • Validation = compilation. No ✓ marked, blank/answer mismatch, stranded prose between shuffled questions… all fail the build, naming the question.
  • Figures, tables, math. Questions are ordinary Typst content — confusion matrices, plots, code blocks, blanks inside equations all work.
  • Tested. 78 pytest cases: determinism, fairness, key correctness, RNG uniformity, 100-question stress papers, 31 exact compile-error messages.

Quick start

git clone https://github.com/nipunbatra/typst-randomize-quiz
cd typst-randomize-quiz
make install-local        # @local/quizforge:0.1.0 available everywhere
make build EXAM=exams/quiz3.typ   # all sets + keys + answer_key.csv
